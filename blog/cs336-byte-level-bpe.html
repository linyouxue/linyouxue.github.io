在学习 Stanford CS336 Lecture 1 之前，我对 tokenizer 的理解基本停留在一句话：

tokenizer 把文本转换成 token ID。

这句话没有错，但它把最关键的过程全部藏了起来：文本为什么要被切分？应该切成字符、单词还是子词？中文、Emoji 和从未见过的字符串怎么办？为什么相同文本在不同模型中会产生不同数量的 token？训练 tokenizer 时，词表又是怎样从 256 个基础字节逐步扩张出来的？

CS336 的第一讲把 tokenization 放在语言模型系统的起点：模型并不直接处理字符串，而是处理整数序列；tokenizer 决定模型看到的“基本单位”是什么。为了真正理解这一步，我没有直接调用现成库，而是把一个带预分词边界的 Byte-Level BPE 训练器拆成若干小任务，从 UTF-8、pair 计数和单轮合并开始，最终组成完整训练循环。

本文使用同一个例子贯穿全部过程：

ab ababab abab

最终目标是让程序自动学习出：

b"a" + b"b"       -> b"ab"
b"ab" + b"ab"     -> b"abab"
b" " + b"abab"    -> b" abab"
b" abab" + b"ab"  -> b" ababab"

1. BPE 是什么

BPE 的全称是 Byte Pair Encoding，通常译为“字节对编码”。

它最初是一种数据压缩方法。用于 tokenizer 时，它的核心思想可以概括为：

从最小单位开始，反复寻找语料中出现频率最高的相邻 token 对，并把这一对合并成一个新 token。

例如，初始 token 序列是：

a b a b a b

其中相邻 pair 的频率为：

(a, b): 3
(b, a): 2

于是把最高频的 (a, b) 合并为新 token ab：

ab ab ab

接下来重新统计：

(ab, ab): 2

再把它合并成 abab：

abab ab

BPE 不是提前规定“一个单词应该如何拆分”，而是让训练语料中的统计规律决定哪些片段值得进入词表。高频片段会逐渐变成较长 token，低频或从未见过的内容仍然可以退回到底层单位表示。

2. BPE 是干什么的

语言模型只能接收整数张量，无法直接处理 Python 字符串。Tokenizer 的任务是在两种表示之间建立可逆映射：

原始文本 <-> token ID 序列

BPE 在这里主要解决四个问题。

2.1 把任意文本转换成整数

例如：

ab

在 UTF-8 中对应：

[97, 98]

训练后，程序可能把这两个基础字节合并成一个新 token：

[256]

模型最终处理的是 [256]，而不是字符串 "ab"。

2.2 缩短序列

纯 byte tokenizer 能表示所有文本，但序列通常较长。BPE 会把频繁出现的字节片段压缩为一个 token。

例如：

[97, 98, 97, 98, 97, 98]

经过两轮学习后可能变成：

[257, 256]

底层内容没有改变，但模型需要处理的位置数减少了。

2.3 在词表大小与序列长度之间折中

纯单词词表容易遇到未登录词，词表也可能极大；纯字符或纯字节词表虽然通用，但序列过长。

BPE 学到的是可变长度子词：

单字节 -> 字符片段 -> 常见子词 -> 高频单词

因此它同时保留了开放词表能力和一定的压缩效率。

2.4 保证可逆

每个新 token 都被定义为两个旧 token 所代表 bytes 的拼接：

vocab[new_token_id] = vocab[left_id] + vocab[right_id]

所以 BPE 只改变 token 边界，不改变原始字节内容。只要按照顺序查表并拼接 bytes，就能恢复训练前的文本。

3. 为什么实现 Byte-Level BPE

如果直接从 Unicode 字符开始构建词表，需要面对数量庞大的字符集合和未见字符问题。Byte-Level BPE 先把字符串编码为 UTF-8 bytes，再从全部 256 种单字节出发：

vocab = {
    0: bytes([0]),
    1: bytes([1]),
    ...
    255: bytes([255])
}

由于任意 UTF-8 文本都能表示成 0～255 的字节序列，初始词表天然覆盖任意字符串，包括：

  • 英文；
  • 中文；
  • Emoji；
  • 代码；
  • 生僻字符；
  • 训练语料中从未出现过的文本。

需要区分两个概念：

bytes：token 实际代表的底层字节内容
token ID：训练过程中用来引用该内容的整数编号

训练开始时，二者数值恰好都在 0～255；但第一次 merge 之后，新 token ID 会从 256 开始，因此训练中的 token 序列不能一直使用 bytes 保存，而要使用 list[int] 或 tuple[int, ...]。

4. 为什么还需要预分词

最朴素的 BPE 可以直接在整篇文本上统计 pair，但现代 tokenizer 通常先进行 pretokenization。

以本文示例为例：

ab ababab abab

使用正则表达式预分词后得到：

[
    b"ab",
    b" ababab",
    b" abab"
]

这里后两个 pretoken 保留了前导空格。BPE 只在每个 pretoken 内部合并，不允许跨边界建立 pair。

这样做有两个直接效果：

  1. 标点、单词、数字和空白之间的边界更加稳定；
  2. 训练时不会学到大量跨边界且缺乏复用价值的片段。

完整流程因此变成：

Python str
    |
    v
正则预分词
    |
    v
每个片段编码为 UTF-8 bytes
    |
    v
聚合相同 pretoken
    |
    v
统计内部相邻 pair
    |
    v
选择最高频 pair
    |
    v
创建新 token 并全局合并
    |
    v
重新统计，进入下一轮

5. 从任务拆解到完整实现

下面按照我实际完成任务的顺序，把最终程序中的每个函数单独拆开。

5.1 pretokenize：预分词并编码为 UTF-8 bytes

函数目标

输入 Python 字符串：

"ab ababab abab"

输出：

[b"ab", b" ababab", b" abab"]

代码如下：

import regex

PRETOKEN_PATTERN = regex.compile(
    r"""'(?:[sdmt]|ll|ve|re)| ?\p{L}+| ?\p{N}+| ?[^\s\p{L}\p{N}]+|\s+(?!\S)|\s+"""
)


def pretokenize(text):
    pretokens = []

    for match in PRETOKEN_PATTERN.finditer(text):
        piece = match.group()

        if piece != "":
            pretokens.append(piece.encode("utf-8"))

    return pretokens

逐步变化

输入：

text = "ab ababab abab"

regex.finditer 按原顺序获得：

"ab"
" ababab"
" abab"

逐个调用：

piece.encode("utf-8")

得到：

b"ab"
b" ababab"
b" abab"

最终返回：

[b"ab", b" ababab", b" abab"]

Task 问题

1. 为什么先在 str 上使用正则，再编码为 bytes？

正则中的 \p{L}、\p{N} 表示 Unicode 字母和数字类别，它们针对字符语义工作。先把字符串整体编码成 bytes 后，一个中文字符可能变成多个独立字节，正则就无法再按字符类别正确分组。

2. 为什么返回 list[bytes]，而不是一个完整 bytes？

因为每个列表元素代表一个独立 pretoken。后续 BPE 只能在元素内部统计和合并，不能跨两个元素的边界。

3. 为什么不能产生空 pretoken？

空片段不包含任何 token，既不能形成 pair，也没有训练意义。保留它只会增加无效状态和边界处理。

4. 为什么文件被导入时不能执行 input()？

pretokenize 会被其他训练模块导入。若文件在 import 时自动等待输入，整个训练程序就会被阻塞。测试代码应放在：

if __name__ == "__main__":

之下。

5.2 build_pretoken_counts：建立 pretoken 频率表

函数目标

将预分词结果转换为：

dict[tuple[int, ...], int]

其中：

  • 键是 pretoken 对应的 token ID 元组；
  • 值是该 pretoken 在语料中的出现次数。
def build_pretoken_counts(text):
    pretokens = pretokenize(text)
    pretoken_counts = {}

    for piece in pretokens:
        token_tuple = tuple(piece)

        if token_tuple in pretoken_counts:
            pretoken_counts[token_tuple] += 1
        else:
            pretoken_counts[token_tuple] = 1

    return pretoken_counts

逐步变化

b"ab"

遍历 bytes 会得到：

97, 98

所以：

tuple(b"ab") == (97, 98)

同理：

tuple(b" ababab")

得到：

(32, 97, 98, 97, 98, 97, 98)

最终频率表：

{
    (97, 98): 1,
    (32, 97, 98, 97, 98, 97, 98): 1,
    (32, 97, 98, 97, 98): 1
}

Task 问题

1. 为什么不能用 list[int] 作为字典键？

Python 字典的键必须可哈希。list 可以原地修改，因此不可哈希；tuple 不可变，只要其中元素均可哈希，就可以稳定地作为字典键。

counts[(97, 98)] = 1    # 正确
counts[[97, 98]] = 1    # TypeError
2. 为什么要聚合相同 pretoken？

若一个 pretoken 出现 10000 次，不必保存并扫描 10000 份相同序列。只保存一次序列和频率 10000，后续统计 pair 时乘上频率即可得到相同结果。

3. 聚合是否丢失信息？

它会丢失不同 pretoken 在全文中的排列顺序，但 BPE 不允许跨 pretoken 合并，因此训练只需要每种 pretoken 的内部顺序和总频率。对 pair 统计而言，这些信息已经足够。

5.3 count_weighted_pairs：加权统计相邻 pair

函数目标

遍历每个 pretoken 的所有相邻位置，并按该 pretoken 的出现频率累加。

def count_weighted_pairs(pretoken_counts):
    pair_counts = {}

    for pretoken, frequency in pretoken_counts.items():
        for i in range(len(pretoken) - 1):
            pair = (pretoken[i], pretoken[i + 1])

            if pair in pair_counts:
                pair_counts[pair] += frequency
            else:
                pair_counts[pair] = frequency

    return pair_counts

第一轮统计

初始频率表：

{
    (97, 98): 1,
    (32, 97, 98, 97, 98, 97, 98): 1,
    (32, 97, 98, 97, 98): 1
}

分别展开。

(97, 98) 贡献：

(97, 98): 1

(32, 97, 98, 97, 98, 97, 98) 贡献：

(32, 97): 1
(97, 98): 3
(98, 97): 2

(32, 97, 98, 97, 98) 贡献：

(32, 97): 1
(97, 98): 2
(98, 97): 1

合计：

{
    (97, 98): 6,
    (98, 97): 3,
    (32, 97): 2
}

最高频 pair 是：

(97, 98)

即：

(b"a", b"b")

Task 问题

1. 为什么每次发现 pair 要增加 pretoken 的频率，而不是增加 1？

频率表中的一条记录可能代表原文本中的多次出现。若某 pretoken 频率是 100，那么它内部每个相邻 pair 也在语料中出现了 100 份，因此应增加 100。

2. 为什么不统计两个 pretoken 之间的 pair？

预分词已经规定了 BPE 边界。跨边界 pair 即使在原字符串中物理相邻，也不是允许合并的候选。

3. 为什么不直接一直用 bytes 表示 token 序列？

bytes 的每个元素只能在 0～255。BPE 新 token 从 ID 256 开始，所以训练中的序列必须能存储任意整数 ID。

4. 为什么 (a, a, a) 中能统计出两个 (a, a)？

相邻位置有：

位置 0、1
位置 1、2

pair 统计允许重叠，它统计的是候选 pair 出现位置；但执行一轮 merge 时，一个 token 不能参与两次合并，因此实际只能非重叠合并一次。

5. 聚合相同 pretoken 为什么能提高训练效率？

因为程序只扫描每种唯一序列一次，再用 frequency 对结果加权。语料重复度越高，节省的重复遍历和存储越多。

5.4 merge_pair：在单个序列中执行非重叠合并

函数目标

给定：

tokens = [97, 98, 97, 98]
pair = (97, 98)
new_token_id = 256

返回：

[256, 256]

实现：

def merge_pair(tokens, pair, new_token_id):
    merged_tokens = []
    i = 0

    while i < len(tokens):
        has_next_token = i + 1 < len(tokens)

        if has_next_token:
            current_pair = (tokens[i], tokens[i + 1])
        else:
            current_pair = None

        if current_pair == pair:
            merged_tokens.append(new_token_id)
            i += 2
        else:
            merged_tokens.append(tokens[i])
            i += 1

    return merged_tokens

为什么是 i += 2

匹配成功时，当前位置和下一位置已经共同组成一个新 token，两个旧 token 都被消费，所以扫描位置必须跨过 2。

匹配失败时，只保留当前 token，下一位置仍可能与其后面的 token 组成 pair，因此只执行 i += 1。

重叠示例

tokens = [256, 256, 256]
pair = (256, 256)

虽然 pair 计数中有两个重叠位置，但从左向右合并后：

[256, 256, 256]
 └──────┘

得到：

[257, 256]

而不是两个 257。

Task 问题

1. 为什么统计允许重叠，合并却不允许重叠？

统计阶段衡量一种局部模式有多常见；合并阶段则要生成一个合法的新序列。一个旧 token 无法同时属于两个新 token，所以同一轮合并必须选择互不重叠的位置。

2. 为什么返回新列表，而不是原地修改？

创建新列表让扫描逻辑更稳定，不会因序列长度在遍历过程中变化而造成下标错乱，同时也不会修改调用者持有的原始数据。

3. 为什么必须从左向右？

对于存在重叠候选的位置，必须有明确规则才能得到确定结果。从左向右是常见且简单的非重叠选择策略。

5.5 merge_pair_in_counts：把一条 merge 应用于全部 pretoken

函数目标

前一个函数只处理一个 token 序列。训练时需要把同一条 merge 规则应用于所有 pretoken：

def merge_pair_in_counts(pretoken_counts, pair, new_token_id):
    new_counts = {}

    for pretoken, frequency in pretoken_counts.items():
        token_list = list(pretoken)

        merged_list = merge_pair(
            token_list,
            pair,
            new_token_id
        )

        merged_tuple = tuple(merged_list)

        if merged_tuple in new_counts:
            new_counts[merged_tuple] += frequency
        else:
            new_counts[merged_tuple] = frequency

    return new_counts

第一轮变化

合并：

(97, 98) -> 256

原频率表：

{
    (97, 98): 1,
    (32, 97, 98, 97, 98, 97, 98): 1,
    (32, 97, 98, 97, 98): 1
}

变为：

{
    (256,): 1,
    (32, 256, 256, 256): 1,
    (32, 256, 256): 1
}

频率没有变化，变化的是每种 pretoken 的 token 划分。

Task 问题

1. 为什么不能一边遍历原字典，一边替换它的键？

替换字典键实际上需要删除旧键并插入新键，会改变字典结构。Python 通常会抛出：

RuntimeError: dictionary changed size during iteration

因此应构造新的 new_counts。

2. 为什么合并后 frequency 通常不需要重新计算？

BPE 只改变一个文本片段内部的 token 边界，不改变该片段在原语料中出现了多少次。

3. 为什么不同旧 pretoken 合并到同一个新 tuple 时要累加频率？

它们已经变成训练状态完全相同的序列，应当重新聚合。新序列的总频率等于所有来源频率之和。

4. 为什么 new_token_id 必须是从未使用过的新 ID？

一个 token ID 必须唯一指向一种 bytes 内容。复用旧 ID 会让同一个编号同时代表两种内容，导致词表、merge 和解码出现歧义。

5. 为什么 merge_pair 返回的 list 要转回 tuple？

list 便于构造合并结果，但不能作为字典键；频率表的键必须是可哈希的 tuple。

5.6 choose_best_pair：选择最高频 pair

函数目标

BPE 每轮应选择频率最高的 pair。若出现并列，还需要确定性的 tie-breaking 规则：

def choose_best_pair(pair_counts, vocab):
    if len(pair_counts) == 0:
        return None

    best_pair = None

    for pair in pair_counts:
        if best_pair is None:
            best_pair = pair
            continue

        current_frequency = pair_counts[pair]
        best_frequency = pair_counts[best_pair]

        if current_frequency > best_frequency:
            best_pair = pair

        elif current_frequency == best_frequency:
            current_bytes_pair = (
                vocab[pair[0]],
                vocab[pair[1]]
            )

            best_bytes_pair = (
                vocab[best_pair[0]],
                vocab[best_pair[1]]
            )

            if current_bytes_pair > best_bytes_pair:
                best_pair = pair

    return best_pair

本实现规定：

pair 频率相同时，选择对应 bytes 二元组中字典序更大的 pair。

这条规则本身不是 BPE 唯一可能的规定，但必须明确，否则相同语料可能因为遍历顺序不同而得到不同 merge 列表。

Task 问题

1. 为什么不能只写 max(pair_counts, key=pair_counts.get)？

可以用于最简实现，但它没有显式表达并列时如何选择。若要保证结果可复现，应把 tie-breaking 写清楚。

2. 为什么并列时比较 bytes，而不是直接比较 token ID？

新 token ID 取决于先前训练过程。比较 token 所代表的 bytes 更接近比较实际内容，而不是比较临时编号。

3. 为什么空频率表返回 None？

空表表示所有 pretoken 长度都小于 2，已经不存在相邻 pair，训练循环应提前终止。

5.7 show_pretoken_counts：把内部状态转换成可读形式

函数目标

训练内部保存的是 token ID：

(32, 256, 256)

调试时更希望看到：

[b" ", b"ab", b"ab"]

因此增加一个只负责显示的辅助函数：

def show_pretoken_counts(pretoken_counts, vocab):
    readable_counts = []

    for pretoken, frequency in pretoken_counts.items():
        byte_parts = []

        for token_id in pretoken:
            byte_parts.append(vocab[token_id])

        readable_counts.append((byte_parts, frequency))

    return readable_counts

这个函数不参与训练决策，只把 ID 通过 vocab 映射回 bytes，方便观察每轮发生了什么。

Task 问题

1. 为什么不直接把训练状态存成 bytes？

训练状态必须保存 256 以上的新 token ID；显示函数只在需要输出时临时查词表，避免混淆“token ID 序列”和“token 实际内容”。

2. 为什么显示函数不能修改 pretoken_counts？

调试工具不应改变算法状态。否则打开或关闭 show_steps 可能导致训练结果不同。

3. 为什么一个 token 可能显示为不完整 UTF-8 bytes？

Byte-Level BPE 的中间 token 不一定恰好对应完整 Unicode 字符。只有把完整 token 序列对应的 bytes 按顺序拼接后，再统一 UTF-8 解码，才能保证正确恢复文本。

5.8 train_byte_bpe：组织完整训练循环

前面的函数分别完成了预分词、聚合、pair 统计和单轮合并。最后的训练函数负责把它们按顺序连接起来。

核心流程：

def train_byte_bpe(
    text,
    vocab_size,
    min_frequency=1,
    show_steps=False
):
    if vocab_size < 256:
        raise ValueError("vocab_size 不能小于 256")

    if min_frequency < 1:
        raise ValueError("min_frequency 不能小于 1")

    vocab = {}

    for token_id in range(256):
        vocab[token_id] = bytes([token_id])

    merges = []
    pretoken_counts = build_pretoken_counts(text)
    round_number = 1

    while len(vocab) < vocab_size:
        pair_counts = count_weighted_pairs(pretoken_counts)
        best_pair = choose_best_pair(pair_counts, vocab)

        if best_pair is None:
            break

        best_frequency = pair_counts[best_pair]

        if best_frequency < min_frequency:
            break

        new_token_id = len(vocab)

        left_id = best_pair[0]
        right_id = best_pair[1]

        left_bytes = vocab[left_id]
        right_bytes = vocab[right_id]

        vocab[new_token_id] = left_bytes + right_bytes
        merges.append((left_bytes, right_bytes))

        pretoken_counts = merge_pair_in_counts(
            pretoken_counts,
            best_pair,
            new_token_id
        )

        round_number += 1

    return vocab, merges, pretoken_counts

Task 问题

1. 为什么每轮合并后必须重新统计 pair？

一次 merge 会删除旧 pair，也会产生包含新 token 的新 pair。例如：

(a, b, c) -> (ab, c)

原来的 (a,b)、(b,c) 消失，新的 (ab,c) 出现。上一轮的频率表已不再描述当前状态。

2. 为什么新 ID 应只在成功合并后推进？

每个新 ID 都应对应一个实际创建的词表项和一条 merge 规则。若在无 pair 或未达到最低频率时仍推进 ID，就会产生空洞，并破坏 ID 与 merge 次数之间的关系。

3. 为什么 len(vocab) == 256 + len(merges)？

初始词表固定有 256 项。每成功一轮：

  • vocab 增加一个新 token；
  • merges 增加一条规则。

二者始终同步增长。

4. 聚合后的频率表丢失原始顺序，为什么仍足以训练？

训练只统计 pretoken 内部 pair 的总频率，不允许跨 pretoken 合并。每个 pretoken 的内部顺序仍保存在 tuple 中，而不同 pretoken 的全文先后顺序不会改变合法 pair 统计。

5. 为什么词表项的 bytes 拼接能保证文本不变？

定义：

vocab[new_id] = vocab[left_id] + vocab[right_id]

所以合并前后对应的 bytes 完全相等。BPE 改变的是分组方式，不是数据本身。

6. vocab_size 和 min_frequency 分别控制什么？
  • vocab_size 限制最终词表最多有多少项；
  • min_frequency 阻止程序继续合并极低频 pair。

对本文极小示例，最后一轮频率只有 1，因此需要：

min_frequency = 1

6. 这份实现与生产级 tokenizer 的差距

这份代码完整实现了 Byte-Level BPE 的核心训练逻辑，但定位仍是教学实现。

每一轮都会：

  1. 扫描全部 pretoken；
  2. 重新统计全部 pair；
  3. 再扫描全部 pretoken执行合并。

这种写法清晰、容易验证，但在大语料上速度较慢。生产级 tokenizer 通常会：

  • 增量维护 pair 频率；
  • 保存 pair 到出现位置的索引；
  • 使用更紧凑的数据结构；
  • 用 Rust 或 C++ 实现热点路径；
  • 单独处理特殊 token；
  • 保存可直接用于编码的 merge rank；
  • 对正则预分词和批处理进行优化。

但从学习角度看，先实现这个版本很重要。只有明确理解“当前状态是什么、pair 如何统计、merge 如何改变状态”，后续的性能优化才不是黑箱。

# 导入第三方 regex 模块。
# 它支持 \p{L}、\p{N} 等 Unicode 字符类别。
import regex


# 编译预分词正则表达式。
# 该规则会把文本分为英文缩写、字母、数字、标点和空白等片段。
# BPE 只允许在同一个 pretoken 内部合并，不能跨越这些片段的边界。
PRETOKEN_PATTERN = regex.compile(
    r"""'(?:[sdmt]|ll|ve|re)| ?\p{L}+| ?\p{N}+| ?[^\s\p{L}\p{N}]+|\s+(?!\S)|\s+"""
)


def pretokenize(text):
    """
    将 Python 字符串预分词，并把每个片段编码成 UTF-8 bytes。

    参数：
        text：需要处理的 Python 字符串。

    返回：
        list[bytes]，其中每一项都是一个 pretoken。
    """

    # 创建空列表，用于保存最终的 bytes 片段。
    pretokens = []

    # finditer 从左向右返回每一个正则匹配对象。
    for match in PRETOKEN_PATTERN.finditer(text):

        # group() 取得当前匹配到的完整字符串。
        piece = match.group()

        # 正则通常不会产生空匹配，但这里仍然进行保护。
        if piece != "":

            # 先取得完整字符串片段，再把它编码为 UTF-8 bytes。
            encoded_piece = piece.encode("utf-8")

            # 把编码后的 pretoken 加入结果列表。
            pretokens.append(encoded_piece)

    # 空字符串不会产生匹配，因此会自然返回空列表。
    return pretokens


def build_pretoken_counts(text):
    """
    建立 pretoken 频率表。

    返回结构：
        dict[tuple[int, ...], int]

    字典键：
        pretoken 对应的 token ID 元组。

    字典值：
        该 pretoken 在文本中的出现频率。
    """

    # 调用预分词函数，得到 list[bytes]。
    pretokens = pretokenize(text)

    # 创建空字典，用于保存 pretoken 频率。
    pretoken_counts = {}

    # 依次处理每个 pretoken。
    for piece in pretokens:

        # 遍历 bytes 会得到 0～255 的字节整数。
        # 将其转换成 tuple，使它可以作为字典键。
        token_tuple = tuple(piece)

        # 如果该 pretoken 已经出现过，就把频率增加 1。
        if token_tuple in pretoken_counts:
            pretoken_counts[token_tuple] += 1

        # 如果是第一次出现，就把频率初始化为 1。
        else:
            pretoken_counts[token_tuple] = 1

    # 返回聚合后的频率表。
    return pretoken_counts


def count_weighted_pairs(pretoken_counts):
    """
    加权统计所有 pretoken 内部的相邻 token pair。

    返回结构：
        dict[tuple[int, int], int]

    注意：
        不统计两个不同 pretoken 之间的 pair。
    """

    # 创建新的 pair 频率字典。
    pair_counts = {}

    # items() 同时取得 pretoken 元组和它的频率。
    for pretoken, frequency in pretoken_counts.items():

        # 长度为 n 的 pretoken 中有 n - 1 个相邻位置。
        for i in range(len(pretoken) - 1):

            # 取当前位置和下一个位置，组成有顺序的二元组。
            pair = (pretoken[i], pretoken[i + 1])

            # 如果该 pair 已经出现过，就增加整个 pretoken 的频率。
            if pair in pair_counts:
                pair_counts[pair] += frequency

            # 第一次遇到时，初始值也应是 pretoken 的频率。
            else:
                pair_counts[pair] = frequency

    # 返回全部 pair 的加权频率。
    return pair_counts


def merge_pair(tokens, pair, new_token_id):
    """
    在一个 token 序列中，从左向右非重叠合并指定 pair。

    参数：
        tokens：list[int]，原 token 序列。
        pair：tuple[int, int]，需要合并的 token pair。
        new_token_id：合并后使用的新 token ID。

    返回：
        合并后的新 list[int]。
    """

    # 创建新列表，不直接修改传入的原列表。
    merged_tokens = []

    # i 表示当前正在检查的下标。
    i = 0

    # 只要 i 没有越过列表末尾，就继续处理。
    while i < len(tokens):

        # 首先保证当前位置后面还有一个 token。
        has_next_token = i + 1 < len(tokens)

        # 只有存在下一个 token 时，才能组成相邻 pair。
        if has_next_token:

            # 把当前位置的两个 token 组成二元组。
            current_pair = (tokens[i], tokens[i + 1])

        # 如果当前位置已经是最后一个位置，就无法组成 pair。
        else:
            current_pair = None

        # 如果当前位置恰好匹配指定 pair，就执行合并。
        if current_pair == pair:

            # 用一个新的 token ID 替换两个旧 token。
            merged_tokens.append(new_token_id)

            # 两个旧 token 都已经被使用，因此跨过两个位置。
            i += 2

        # 如果当前位置不匹配指定 pair，就保留当前 token。
        else:
            merged_tokens.append(tokens[i])

            # 只处理了一个 token，因此向右移动一个位置。
            i += 1

    # 返回新列表。
    return merged_tokens


def merge_pair_in_counts(pretoken_counts, pair, new_token_id):
    """
    将指定 pair 应用于整个 pretoken 频率表。

    每个 pretoken 独立合并，不允许跨 pretoken 边界。
    """

    # 创建新的频率表，避免在遍历时修改原字典。
    new_counts = {}

    # 遍历每一种唯一 pretoken 及其原频率。
    for pretoken, frequency in pretoken_counts.items():

        # merge_pair 接收 list，因此先把 tuple 转换成 list。
        token_list = list(pretoken)

        # 调用统一的非重叠合并函数。
        merged_list = merge_pair(
            token_list,
            pair,
            new_token_id
        )

        # 字典键不能使用 list，因此重新转换成 tuple。
        merged_tuple = tuple(merged_list)

        # 不同旧 pretoken 可能得到相同的新 tuple。
        # 此时应将它们的频率相加。
        if merged_tuple in new_counts:
            new_counts[merged_tuple] += frequency

        # 如果该新 tuple 首次出现，就保留原 pretoken 的频率。
        else:
            new_counts[merged_tuple] = frequency

    # 返回本轮合并后的新频率表。
    return new_counts


def choose_best_pair(pair_counts, vocab):
    """
    选择本轮需要合并的最高频 pair。

    并列规则：
        频率相同时，选择 bytes 二元组字典序更大的 pair。

    明确规定并列规则，可以保证相同输入每次得到相同结果。
    """

    # 如果频率表为空，说明当前已经没有可合并 pair。
    if len(pair_counts) == 0:
        return None

    # best_pair 暂时设为空。
    best_pair = None

    # 依次检查每一种 pair。
    for pair in pair_counts:

        # 第一次循环时，直接把当前 pair 设为最优 pair。
        if best_pair is None:
            best_pair = pair
            continue

        # 取得当前 pair 的频率。
        current_frequency = pair_counts[pair]

        # 取得目前最优 pair 的频率。
        best_frequency = pair_counts[best_pair]

        # 当前 pair 频率更高时，直接更新。
        if current_frequency > best_frequency:
            best_pair = pair

        # 频率相同时，执行确定性的字节字典序比较。
        elif current_frequency == best_frequency:

            # 找到当前 pair 左 token 代表的 bytes。
            current_left = vocab[pair[0]]

            # 找到当前 pair 右 token 代表的 bytes。
            current_right = vocab[pair[1]]

            # 组成当前 pair 的 bytes 二元组。
            current_bytes_pair = (current_left, current_right)

            # 找到目前最优 pair 左 token 代表的 bytes。
            best_left = vocab[best_pair[0]]

            # 找到目前最优 pair 右 token 代表的 bytes。
            best_right = vocab[best_pair[1]]

            # 组成目前最优 pair 的 bytes 二元组。
            best_bytes_pair = (best_left, best_right)

            # 字典序更大的 pair 胜出。
            if current_bytes_pair > best_bytes_pair:
                best_pair = pair

    # 返回最终选中的整数 ID pair。
    return best_pair



def show_pretoken_counts(pretoken_counts, vocab):
    """
    把 pretoken 频率表转换成更容易阅读的形式。

    参数：
        pretoken_counts：
            dict[tuple[int, ...], int]

        vocab：
            dict[int, bytes]

    返回：
        list[tuple[list[bytes], int]]

    贯穿示例第一轮训练前：

        pretoken_counts = {
            (97, 98): 1,
            (32, 97, 98, 97, 98, 97, 98): 1,
            (32, 97, 98, 97, 98): 1
        }

    转换后：

        [
            ([b"a", b"b"], 1),
            ([b" ", b"a", b"b", b"a", b"b", b"a", b"b"], 1),
            ([b" ", b"a", b"b", b"a", b"b"], 1)
        ]

    这个函数只用于显示训练过程，不修改 vocab 或 pretoken_counts。
    """

    # 创建结果列表。
    readable_counts = []

    # 遍历每一种 pretoken 及其频率。
    for pretoken, frequency in pretoken_counts.items():

        # 保存当前 pretoken 中每个 token ID 所代表的 bytes。
        byte_parts = []

        # 按照 pretoken 内部的原顺序查找词表。
        for token_id in pretoken:
            byte_parts.append(vocab[token_id])

        # 把可读的 bytes 列表及其频率加入结果。
        readable_counts.append((byte_parts, frequency))

    # 返回仅用于显示的结果。
    return readable_counts


def train_byte_bpe(
    text,
    vocab_size,
    min_frequency=1,
    show_steps=False
):
    """
    训练带预分词边界的 byte-level BPE。

    贯穿示例：

        text = "ab ababab abab"
        vocab_size = 260
        min_frequency = 1

    正则预分词结果：

        [
            b"ab",
            b" ababab",
            b" abab"
        ]

    转换成初始 token ID 元组：

        {
            (97, 98): 1,

            (32, 97, 98, 97, 98, 97, 98): 1,

            (32, 97, 98, 97, 98): 1
        }

    其中：

        32 -> b" "
        97 -> b"a"
        98 -> b"b"

    第一轮：

        最高频 pair：
            (97, 98)

        出现频率：
            6

        创建：
            256 -> b"ab"

        频率表变为：

            {
                (256,): 1,
                (32, 256, 256, 256): 1,
                (32, 256, 256): 1
            }

    第二轮：

        最高频 pair：
            (256, 256)

        出现频率：
            3

        创建：
            257 -> b"abab"

        频率表变为：

            {
                (256,): 1,
                (32, 257, 256): 1,
                (32, 257): 1
            }

    第三轮：

        最高频 pair：
            (32, 257)

        出现频率：
            2

        创建：
            258 -> b" abab"

        频率表变为：

            {
                (256,): 1,
                (258, 256): 1,
                (258,): 1
            }

    第四轮：

        最高频 pair：
            (258, 256)

        出现频率：
            1

        创建：
            259 -> b" ababab"

        最终频率表：

            {
                (256,): 1,
                (259,): 1,
                (258,): 1
            }

    最终三个 pretoken 分别表示：

        256 -> b"ab"
        259 -> b" ababab"
        258 -> b" abab"

    因此，按照原始顺序拼接后仍为：

        b"ab" + b" ababab" + b" abab"

        == b"ab ababab abab"
    """

    # byte-level BPE 至少需要包含 256 个基础字节 token。
    if vocab_size < 256:
        raise ValueError("vocab_size 不能小于 256")

    # pair 的最低合并频率必须至少为 1。
    if min_frequency < 1:
        raise ValueError("min_frequency 不能小于 1")

    # 创建空词表。
    vocab = {}

    # 建立 0～255 共 256 个基础 token。
    for token_id in range(256):

        # ID 为 i 的基础 token 对应单字节 bytes([i])。
        vocab[token_id] = bytes([token_id])

    # 示例中：
    #
    # vocab[32] == b" "
    # vocab[97] == b"a"
    # vocab[98] == b"b"

    # 创建空列表，按顺序保存训练得到的合并规则。
    merges = []

    # 对原字符串进行预分词，并聚合相同的 pretoken。
    #
    # 示例得到：
    #
    # {
    #     (97, 98): 1,
    #     (32, 97, 98, 97, 98, 97, 98): 1,
    #     (32, 97, 98, 97, 98): 1
    # }
    pretoken_counts = build_pretoken_counts(text)

    # 第一轮的编号为 1。
    round_number = 1

    # 根据参数决定是否输出训练过程。
    if show_steps:
        print("原字符串：", text)

        # 示例输出：
        # [b'ab', b' ababab', b' abab']
        print("预分词结果：", pretokenize(text))

        print("初始 token 状态：")
        print(show_pretoken_counts(pretoken_counts, vocab))
        print("-" * 70)

    # 词表未达到目标大小时，继续执行训练。
    while len(vocab) < vocab_size:

        # 每一轮都根据当前 token 状态重新统计相邻 pair。
        #
        # 示例第一轮得到：
        #
        # {
        #     (97, 98): 6,
        #     (98, 97): 3,
        #     (32, 97): 2
        # }
        pair_counts = count_weighted_pairs(pretoken_counts)

        # 从 pair_counts 中选择频率最高的 pair。
        #
        # 示例第一轮：
        # best_pair = (97, 98)
        best_pair = choose_best_pair(pair_counts, vocab)

        # 如果不存在相邻 pair，说明所有 pretoken 都只剩一个 token。
        if best_pair is None:
            break

        # 读取最高频 pair 的频率。
        #
        # 示例第一轮：
        # best_frequency = 6
        best_frequency = pair_counts[best_pair]

        # 最高频 pair 也达不到最低频率时停止训练。
        if best_frequency < min_frequency:
            break

        # token ID 始终连续，因此当前词表长度就是下一个新 ID。
        #
        # 四轮中依次得到：
        # 256、257、258、259
        new_token_id = len(vocab)

        # 取得最高频 pair 左侧的 token ID。
        left_id = best_pair[0]

        # 取得最高频 pair 右侧的 token ID。
        right_id = best_pair[1]

        # 根据词表查出左 token 实际代表的 bytes。
        left_bytes = vocab[left_id]

        # 根据词表查出右 token 实际代表的 bytes。
        right_bytes = vocab[right_id]

        # 按照原顺序拼接左右两部分 bytes。
        #
        # 第一轮：
        # b"a" + b"b" == b"ab"
        #
        # 第二轮：
        # b"ab" + b"ab" == b"abab"
        #
        # 第三轮：
        # b" " + b"abab" == b" abab"
        #
        # 第四轮：
        # b" abab" + b"ab" == b" ababab"
        new_token_bytes = left_bytes + right_bytes

        # 将新 token 加入词表。
        vocab[new_token_id] = new_token_bytes

        # 保存本轮的 bytes pair。
        #
        # 四轮结束后：
        #
        # merges = [
        #     (b"a", b"b"),
        #     (b"ab", b"ab"),
        #     (b" ", b"abab"),
        #     (b" abab", b"ab")
        # ]
        merges.append((left_bytes, right_bytes))

        # 输出本轮选择和创建的新 token。
        if show_steps:
            print("第", round_number, "轮")
            print("最高频 ID pair：", best_pair)
            print("pair 频率：", best_frequency)
            print("对应 bytes pair：", (left_bytes, right_bytes))
            print(
                "创建新 token：",
                new_token_id,
                "->",
                new_token_bytes
            )

        # 在所有 pretoken 内执行本轮非重叠合并。
        #
        # 第一轮之后：
        #
        # {
        #     (256,): 1,
        #     (32, 256, 256, 256): 1,
        #     (32, 256, 256): 1
        # }
        pretoken_counts = merge_pair_in_counts(
            pretoken_counts,
            best_pair,
            new_token_id
        )

        # 输出合并后的状态。
        if show_steps:
            print("合并后的 token 状态：")
            print(show_pretoken_counts(pretoken_counts, vocab))
            print("-" * 70)

        # 只有本轮成功创建并应用了新 token 后，轮数才增加。
        round_number += 1

    # 返回最终词表、合并规则和训练后的频率表。
    return vocab, merges, pretoken_counts

8. 总结

完成 CS336 Lecture 1 和这一组 BPE 任务后，我对 tokenizer 的理解发生了明显变化。

以前我把它看成一个 API：

text -> token IDs

现在更准确的理解是：

Tokenizer 是一个在可逆性、开放词表能力、词表规模和序列长度之间进行折中的表示系统。

而 BPE 的训练过程，本质上是一种基于语料统计的贪心词表构造：

  1. 从足以表示任意文本的基础 bytes 出发；
  2. 通过预分词限定合法合并边界；
  3. 聚合相同 pretoken，减少重复计算；
  4. 加权统计全部相邻 pair；
  5. 选择最高频 pair；
  6. 创建代表 bytes 拼接结果的新 token；
  7. 非重叠地更新所有序列；
  8. 重新统计并继续下一轮。

算法本身并不复杂，真正容易出错的是数据表示与边界条件：

  • str、bytes 和 token ID 不是同一层概念；
  • bytes 不能保存 256 以上的新 token ID；
  • pair 统计允许重叠，单轮合并不允许重叠；
  • pretoken 可以聚合，但内部 token 顺序不能丢失；
  • 新 token ID 必须唯一；
  • 合并后必须重新统计 pair；
  • 词表中的 bytes 拼接必须保持原文可逆。

把这些问题逐个实现以后，BPE 不再只是“反复合并最高频字符对”这一句定义，而是一个可以实际运行、检查和扩展的 tokenizer 训练过程。
